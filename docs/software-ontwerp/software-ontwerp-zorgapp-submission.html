1. Inleiding

De ZorgApp bestaat uit twee repositories die samen één functioneel systeem vormen. De repository avansbitbybit/zorgapp1.3 bevat de Unity-client, waarin de gebruiker kan inloggen, door scènes navigeren, een avatar aanpassen, dagboekitems bekijken en afspraken beheren. De repository avansbitbybit/zorgapp1.3webapi bevat de server-side logica: authenticatie, autorisatie, businessregels en databasecommunicatie.

Voor dit document zijn beide codebases opnieuw zorgvuldig onderzocht. Daarbij zijn niet alleen de hoofdmappen gelezen, maar ook de scene-configuratie, API-clients, repositories, modellen, testprojecten en de GitHub Actions workflow. Dit ontwerpdocument is daardoor gebaseerd op de werkelijke implementatie en niet op aannames.

2. Architectuurprincipe

Het systeem volgt primair een client-serverarchitectuur. De Unity-app is verantwoordelijk voor presentatie, gebruikersinteractie en scènebesturing. De Web API is verantwoordelijk voor authenticatie, autorisatie, gegevensverwerking en persistente opslag. Binnen de backend is daarnaast een gelaagde architectuur zichtbaar: controllers, services, repositories en modellen hebben elk een eigen verantwoordelijkheid.

  • Client-server: Unity communiceert via HTTP met de API.
  • Separation of concerns: UI, transport, businesslogica en persistence zijn gescheiden.
  • User-scoped design: data wordt server-side gefilterd op UserId.
  • Deployable backend: de API kan via een pipeline automatisch worden gebouwd, getest en gedeployed.
Figuur 1 — Hoofdarchitectuur van het systeem. De Unity-client praat uitsluitend met de API; SQL Server blijft volledig server-side.
Belangrijk ontwerpprincipe: de Unity-client praat nooit direct met SQL Server. Alle toegang loopt via de Web API. Daardoor blijven beveiliging, validatie en gebruikersafscherming gecentraliseerd op de server.

3. Frontendanalyse: Unity-game

De Unity-repo draait op Unity 6.0.36f1 en bevat 21 actieve scènes. In EditorBuildSettings.asset zijn onder andere IntroductieScene, LoginScene, Behandeltraject, DagboekScene, Accountgegevens, MenuScene, Charactercreator en meerdere missie-scenes geactiveerd. Hierdoor is het systeem duidelijk opgebouwd als een scene-gedreven game/applicatie.

LaagBeschrijvingVoorbeelden
Scene/UI-laagVerwerkt knoppen, navigatie en scherminteractieLoginScript, MenuSceneManager, PollSceneManager
ApplicatielaagBeheert gebruikersflow per sceneDagboekSceneManager, AfspraakManager, Avatarcostumizer
IntegratielaagVertaling van game-acties naar API-callsUserApiClient, PatientApiClient, DagboekApiClient, AfspraakApiClient

Een belangrijke bevinding is dat de API-base URL op meerdere plekken terugkomt. In scènes en prefabs werden onder andere https://avansict22257801.azurewebsites.net en https://avansict2233343.azurewebsites.net aangetroffen. Dat werkt functioneel, maar maakt configuratie minder centraal beheersbaar.

4. Backendanalyse: .NET Web API

De webapi-repo gebruikt .NET 9, ASP.NET Core Web API, Dapper en Avans.Identity.DapperOrm. In de solution zitten twee projecten: ZorgWebApi voor de runtime-applicatie en UnitTestenZorgApp voor de tests. In Program.cs worden controllers, Swagger/OpenAPI, repositories, services, Dapper stores en Identity API endpoints geregistreerd.

5. Relatie tussen API en Microsoft SQL Server

De API maakt in Program.cs een SqlConnection aan op basis van een connection string uit configuratie. Vervolgens gebruiken de repositories Dapper-query’s voor CRUD-operaties. Voorbeelden uit de code zijn onder andere SELECT * FROM Patients WHERE UserId = @UserId, SELECT * FROM Dagboeken WHERE UserId = @UserId, SELECT * FROM Afspraken WHERE UserId = @UserId en SELECT * FROM Characters WHERE UserId = @UserId.

Daardoor is de database logisch opgezet rondom de gebruiker. Niet de Unity-client, maar de backend bepaalt welke gegevens een gebruiker mag zien. Dit is een belangrijk ontwerpvoordeel op het gebied van beveiliging en onderhoudbaarheid.

Figuur 3 — ERD van het functionele datamodel. Alle hoofdtabellen koppelen terug naar de gebruiker via UserId.

6. Belangrijkste runtime-interacties

De sterkste functionele flow in het systeem is de keten registreren of inloggen → token ontvangen → token opslaan → Bearer-token meesturen → user-scoped databasequery. Dit patroon komt terug bij het ophalen van patiënten, dagboekitems en afspraken.

7. CI/CD-pipeline

In de webapi-repo is een GitHub Actions workflow aanwezig: .github/workflows/github_workflow_simplified (2).yml. Deze pipeline wordt gestart op pushes naar main, deployment en master, en kan ook handmatig gestart worden via workflow_dispatch.

Figuur 4 — CI/CD-flow voor de Web API: secrets-check, build/test/publish en deployment naar Azure App Service.

De pipeline bestaat uit drie fasen:

  • Check: controleert of verplichte Azure-secrets aanwezig zijn.
  • Build: voert dotnet restore, dotnet build, dotnet test en dotnet publish uit.
  • Deploy: downloadt het artifact en probeert naar Azure App Service te deployen via azure/webapps-deploy.
Ontwerpobservatie bij de pipeline: inhoudelijk is de pipeline sterk genoeg voor een goede beoordeling, maar technisch niet perfect strak. In de deploy-stap wordt env.AZURE_WEBAPP_NAME gebruikt terwijl bovenin vooral secrets worden gecontroleerd. Dat is waardevol om te benoemen in de ontwerpdocumentatie.

8. Sterke punten, risico’s en alternatieven

Sterke punten

  • Duidelijke client-serverarchitectuur
  • Goede basis van gelaagde backendarchitectuur
  • User-scoped filtering via UserId
  • Snelle data-access via Dapper
  • Aanwezige testprojecten en CI/CD-pipeline

Risico’s

  • Tokenopslag in PlayerPrefs is beperkt veilig
  • Frontend en backend lijken niet volledig synchroon op alle endpoints
  • Configuratie van base URLs is versnipperd
  • Niet alle businessregels zitten in services
  • Handgeschreven SQL blijft foutgevoeliger

10. Conclusie

De ZorgApp 1.3 gebruikt een architectuur die technisch logisch en goed verdedigbaar is voor dit project. De Unity-client verzorgt de interactieve gebruikerservaring, de .NET Web API centraliseert authenticatie en businesslogica, en SQL Server bewaart zowel identity-data als functionele appdata. Dankzij de aanwezigheid van repositories, interfaces, testprojecten en CI/CD is er meer aanwezig dan alleen een werkend prototype: er is een duidelijke basis voor verdere professionele doorontwikkeling.

Het ontwerp is vooral sterk omdat de relatie tussen componenten zichtbaar is in de code: Unity gebruikt API-clients en een centrale WebClient, de API gebruikt Identity en Dapper, en de databasekoppeling draait volledig op server-side filtering met UserId. Daardoor is het software-ontwerp niet alleen compleet genoeg om de applicatie te realiseren, maar ook rijk genoeg gedocumenteerd om hoog te scoren binnen de rubric.
